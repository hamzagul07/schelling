## 3. Integrity audit

**Ledger hashes recheck.** All four sealed SHA-256 digests in `FORECASTS.md` match record files
present in `runs/` (`shasum -a 256` of each `runs/Q-2026-USIRAN-STAGE2*.json`, grepped against the
ledger): `aece91bd…` (v1 challenge), `c87d91ae…` (v1 compromise), `3bc97cd4…` (v2 challenge),
`d55ffc3e…` (v2 compromise). The `runs/` files are gitignored (`git check-ignore runs` → `runs`), as
designed for commit-reveal.

**`schelling verify` on all four sealed records (fresh, 2026-07-22):**

|---|---|---|---|---|
| v1 challenge (`…45d931c6cd91`) | PASS | **FAIL** | PASS (34.576340) | **FAILED** |
| v1 compromise (`…2cbb0bc624f3`) | PASS | PASS | PASS (41.635547) | VERIFIED |
| v2 challenge (`…d4441652019a`) | PASS | PASS | PASS (29.407030) | VERIFIED |
| v2 compromise (compromise `…d4441652019a`) | PASS | PASS | PASS (39.443470) | VERIFIED |

**HEADLINE DISCREPANCY.** The **v1 challenge record fails `verify`**: its stored `inputs_hash`
(`45d931c6cd91…`) does not match a fresh recompute from its own embedded game (`2cbb0bc624f3…`).
Crucially, the recompute equals the v1 *compromise* record's hash (`inputs_hash` excludes the model
field, so the two v1 records share a game), and the **determinism check still passes** — re-solving
the embedded game reproduces `34.576340` exactly, and the **ledger-match passes** (the sealed bytes
are intact). Interpretation: the sealed forecast is real and reproducible; what is stale is the
record's own stored content-address label, because the canonicalization that feeds `inputs_hash`
drifted after this record was created (the v1 compromise record, created later, stores the current
hash). Consequence: an outside auditor running `schelling verify` today gets a **FAILED** on one of
the four flagship forecasts. That is a headline, not a footnote.

**OpenTimestamps proofs — ABSENT for the real ledger.** `ledger-proofs/` contains **only `README.md`**
(`ls -la ledger-proofs/`). There is **no `.ots` proof for the committed `FORECASTS.md`**. The four
seals were made in Session 12, before the OpenTimestamps feature existed (Session 17, D17.2), and the
ledger has not been re-stamped since. So the sealed commitments currently rest on **git history alone**
for their timing, not on an external Bitcoin anchor. The capability works — a fresh `seal` on a scratch
ledger produced a real proof (`…/proofs/L.md-3e96c6a497d1.ots`, `ots` v0.7.2, network reachable) — but
it has **not been applied to the artifact that matters**.

**Grading rubric.** `GRADING-Q-2026-USIRAN-STAGE2.md` is committed (`git log`: `8591c3a`, then
`0c2e6fa` for the ratified edits), dated **2026-07-22, pre-resolution** (resolution 2026-08-31). Its
embedded machine-readable `resolution_rubric` JSON **validates against the `ResolutionRubric` schema**

records the rubric lives in this committed doc rather than in the sealed game, because embedding it in
the game would change the record bytes and break the seal — the rubric is excluded from `inputs_hash`

## 5. Comparative standing (strictly from the repo)

### Defensibly rare/unique — verifiable at HEAD today

- **A complete, open, deterministic BDM/Policon implementation.** The manuscript's own related-work
  basis (`paper/draft/01-introduction.md`) states no complete public implementation exists — the
  Scholz–Calbert–Smith reconstruction reproduced examples but shipped no tool, and the commercial
  successor (Senturion) is proprietary. This repo reproduces the published reference case (9.530, §2)
  and enforces byte-determinism (seeds fixed; dataset SHA-pinned in `BACKTEST.md`; 279 tests hold the
  line). **Defensible now.**
- **Forecast-integrity apparatus rarely seen in this literature:** a commit-reveal SHA-256 ledger,
  a one-command `verify` (recompute + re-solve), a pre-registered grading rubric, and OpenTimestamps
  wiring. The *design* is real and the commands run (§1, §3). **Defensible as engineering** (with the
  §3 caveats that one record fails verify and the real ledger has no proof yet).
- **Pre-registration with cryptographic/commit-order proof** (`3294081` before `8ea92b0`, §2). Rare in
  applied political-forecasting work. **Defensible now.**
- **A genuine noise-floor ceiling result** (oracle gap −0.84, §2) — a substantive, reproducible
  negative finding. **Defensible now.**

### Claimed strengths that are UNPROVEN until 2026-09-01 or an 8-case verdict — do not conflate with the above

- **That the model earns its keep out-of-sample.** The US-Iran ledger is sealed but **ungraded**
  (§3–4). No forward claim is demonstrated.
- **That the model works in coercive settings** — its designed home turf. The coercive head-to-head
  is built but runs on **2 out-of-domain cases with no verdict** (§4.4). Entirely unproven.
- **The "cannot be backdated" anchoring guarantee for the actual forecasts** — true in design, but no
  proof exists for the committed ledger yet (§3).
- **"Track record" / "living document" credibility** — zero graded cycles; ~1-day repo (§4.9–10).
